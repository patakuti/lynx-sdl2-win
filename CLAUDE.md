- 注意事項
  - ソースコードを修正する場合は、main branchで作業しないこと。
  - ソースコードを修正する場合は、小手先の対策ではなく、
    本来あるべき設計、簡潔で確実、安全な設計を優先すること。
  - 大きな修正をする場合は、必ず事前に確認すること。
  - ソースコードを修正した場合は、必ずドキュメントを修正し、
    実装とドキュメントの内容に齟齬がないようにすること。
    - README.md, 02_design.md は特に注意すること。
    - 日付付きのファイル、ディレクトリに保存されたファイルは一時的なものなので除外。
  - ソースコードを修正した場合は、こまめにcommitすること。
  - ソースコードを修正した場合は、必ずbuildすること。
  - Gitのコミットメッセージ, PRは必ず英語で書く
  - PRはupstreamではなく、fork先 (patakuti/lynx-sdl2-win) に作成すること
  - ブランチ構成
    - origin/master: upstream (ThomasDickey/lynx-snapshots) の snapshot 履歴のみ。fork 独自の変更は含まない
    - origin/main: fork 独自の開発ブランチ。PRの base は main を指定すること
    - feature branch は main から作成すること

- ビルド手順
  - configure し直すときは、必ず make distclean してから configure すること
    - stale なオブジェクトファイルが残るとリンクエラーや実行時不具合の原因になる

- 禁止事項
  - 要件なしで設計、計画NG
  - 設計、計画なしで実装NG
  - プラットフォーム固有の動作を仮定に基づいて実装NG
    - 必ず実際の動作を計測・確認してから実装すること
  - 推測に基づくデバッグNG
    - 必ずログやテストツールで実際の値を確認すること
    - 推測による修正の繰り返しは時間の無駄
